Как устроена DLL для античита (текстовое описание)
DLL (Dynamic-Link Library) для античита — это, по сути, набор функций, которые игра или её лаунчер загружает в своё адресное пространство для самодиагностики и защиты. Представьте это как инъекцию "охранника" прямо в тело процесса.

Вот основные компоненты, которые такой "охранник" (DLL) может содержать:

1. Точка входа (DllMain)
Это первая функция, которая вызывается, когда DLL загружается в процесс. В контексте античита здесь происходит инициализация защиты.

Что там происходит (текстом): При загрузке (DLL_PROCESS_ATTACH) создается отдельный поток мониторинга. Этот поток не зависит от основного потока игры и работает в фоне. Важно, чтобы в DllMain не было сложных операций (например, бесконечных циклов или долгих вызовов), так как это может заблокировать загрузку всей игры. Поэтому создается новый поток, а сама DllMain быстро завершает работу.

2. Поток мониторинга (The Monitoring Thread)
Это сердце античита. Функция, которая работает в бесконечном цикле и выполняет проверки с определенной периодичностью.

Задачи потока:

Сканирование памяти: Поток может проходить по областям памяти, выделенным для процесса игры, и искать "сигнатуры" — уникальные последовательности байтов, характерные для известных читов. Если сигнатура найдена в памяти, значит, чит-код внедрен.

Проверка открытых дескрипторов: Поток может проверять, не пытается ли другая программа открыть процесс игры с особыми привилегиями (например, на чтение/запись памяти). Такие попытки часто делают чит-программы (вроде Cheat Engine).

Обнаружение отладчиков: Простая проверка IsDebuggerPresent() или более сложные методы, чтобы понять, не запущена ли игра под управлением отладчика (это классический признак реверс-инжиниринга).

Сканирование окон: Поток может искать в системе окна с названиями, содержащими "Cheat Engine", "ArtMoney", "WPE Pro" и т.д. Если такое окно есть — выносится вердикт о наличии чита.
